Team Tack

Documented Coding Standards

When building tack we adhered to a strict style throughout both our client and server codebases. Due to the syntactic similarities between Ruby and CoffeeScript, we were able to maintain the exact same standards across both codebases.

Indentation, Line Endings, & Whitespace

In both our client and server codebases we use two-space tab indentation, in lieu of four-space or TAB character indentation. While there are readability advantages to using a wider intendation style in lower level languages with relatively verbose syntax, the intentation-based syntax of CoffeeScript and Ruby lends itself it two-space tab indentation.

Additionally, we use the standard UNIX LF line endings throughout the entire codebase. This is a nuanced issue - computers running the Windows operating system use a different line ending character than UNIX based systems. This can cause problems when committing new code or comparing code in merge requests. Often times the system-wide setting overrides all line-endings in a codebase when a commit is issued - thus changing every file. In the interest of consistency and eliminating redundant frequent file changes, we elected to use the standard UNIX LF line ending.

Lastly, we maintain a standard for whitespace in our files. Trailing whitespace is trimmed in all files, and an empty, new line shoud be inserted at the end of every file in the codebase.

EditorConfig

How can we maintain these standards consistently across our codebases and development environments? Our indentation, line ending, and whitspace conventions can make the on-boarding process more difficult. To alleviate this concern we use EditorConfig.

EditorConfig is an established, OS-agnostic standard that most modern text editors interface with. Rather than modifying the user-level settings in each text-editor our team uses, we defined a .editorconfig file in the repository root to ensure our editors behave consistently when working in our codebases. Here's an example

root = true

[*]
indent_style = space
indent_size = 2
end_of_line = lf
charset = utf-8
trim_trailing_whitespace = true
insert_final_newline = true

The benefit of using .editorconfig files is that code standards can be treated as code - this eliminates the need for manual configuration of text editors on different operating systems.

General Purpose Comments

Both the client and server codebases follow a basic convention for comment style. Each single-line comment is denoted with a #, followed by a space, before the comment's text is written. The text should begin with a capital letter for readability, unless the comment is a continuation of the comment from the line above. For example:

#this is a bad comment

# this is a bad comment

# This is a good comment

# This is a good comment,
# and this continues it nicely.

We do not use multi-line comments in our codebases. The reasoning behind this is that two-fold. First, only using single-line comments affords us more stylistic consistency between the two codebases. Secondly, most text editors support a comment/uncomment hotkey ('CTRL' + '/') that creates a single-line comments for each line currently selected with the cursor. This is a very helful hotkey, however it does not support multi-line comment syntax, and thus we made the decision to only use single-line comment syntax.

Classes

Class documentation - what the majority of our documentation consists of, adheres to the following conventions regarding class, attribute, and method definitions.

# MyView Class documentation goes above the definition
# MyView is responsible for...

  template:   require './template' # Single-line attribute comments follow definition (if necessary)

  # MyView.prototype.myMethod documentation goes above the definition

The same conventions apply to our server-side codebase:

  field :name, type: String # Single-line attribute comments follow definition (if necessary)

ToDos & Questions

Due to the high frequency of commits issued to our codebases (commit small, commit often), there are often small issues that go unresolved or ignored in favor of focusing on implementing larger components. We use a specific comment syntax to effectively track and locate these issues in both codebases - # TODO - ....

Additionally, we regularly encounter small uncertainties while writing code. These questions are not inquisitive of the broader design choices outlined in our specification, but rather of small, implementation-level uncertainties we encounter. Similarly to # TODOs, these questions often go temporarily unresolved in favor of implementing larger components. We use a specific syntax to track and locate these comments - # QUESTION - ....

Let's take a look at an example:

# person.rb - Ruby file (documentation comments ommited for brevity)

# ...
class Person

  # Attributes & Relations
  field :name, type: String
  has_and_belongs_to_many :friends, class_name: Person

  # TODO - this method needs documentation
  def myName
    return self.name
  end

  # QUESTION - should we cache this value on the model as an attribute?
  # Calculating this on each request may not scale gracefully.
  def friend_count
    return self.friends.count
  end

The # TODO and # QUESTION syntax enables agile development teams to move quickly, allowing the developer to focus on solving larger problems, while simultaneously creating small and digestable issues that can be resolved by themselves or assigned to other team members.
